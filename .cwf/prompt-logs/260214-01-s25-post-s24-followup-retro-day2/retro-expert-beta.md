### Expert beta: Kent Beck

**Framework**: 작은 안전한 단계(small safe steps)와 Tidy First를 통해 결합도를 낮추고(cohesion/coupling), 빠른 피드백 루프로 변경 비용을 관리하는 점진적 설계 접근.
**Source**: Kent Beck, *Extreme Programming Explained: Embrace Change* (2nd ed.); Kent Beck, *Tidy First?*; Kent Beck, *Implementation Patterns*.
**Why this applies**: 이번 세션은 기능 추가보다 문서/규칙/훅/설치 경계를 재정렬해 변화 비용을 낮추는 작업이 중심이었다. 이는 "지금 구조를 정리하면 이후 변경이 싸진다"는 Beck식 경제성 판단과 직접 맞닿아 있다.

이번 세션에서 잘한 점은, 국소 수정에서 전체 하드닝으로 전환할 때도 한 번에 뒤엎지 않고 반복 리팩터링으로 신호 대 잡음을 개선했다는 것이다. AGENTS와 문서를 여러 라운드로 정리해 중복을 줄인 선택은 Tidy First 관점에서 타당하다. 코드가 아니라 운영 지침이라도, 읽기 비용이 낮아지면 이후 의사결정 속도와 정확도가 올라간다. 즉 이번 정리는 "미래 변경의 마찰"을 선제적으로 줄인 투자였다.

또한 lint 규칙, pre-commit/pre-push 게이트, setup 기반 훅 프로파일 설치로 정책을 실행 가능한 피드백으로 바꾼 결정이 좋았다. XP의 핵심은 빠르고 신뢰 가능한 피드백인데, 팀이 반복 위반을 개인 주의사항이 아니라 자동 검증으로 전환한 것은 매우 Beck스럽다. 특히 날짜 롤오버 혼선을 프로토콜로 명시하고 결정론적 체크 대상으로 포함한 것은 "헷갈리는 규칙은 시스템이 대신 확인하게 하라"는 설계 태도에 부합한다.

개선할 점은 변경 단위의 경계다. 세션 요약대로 반복 충돌이 경계 문제라면, 다음 단계는 "정리(tidying) 변경"과 "행동/정책 변경"을 더 엄격히 분리해 결합도를 낮추는 것이다. 문서 내비게이션 개선, 훅 게이트 강화, setup 설치 정책이 하나의 흐름으로 묶일 때 추적성은 좋아질 수 있지만, 회귀 원인 분리는 어려워진다. Beck 관점에서 이는 기술적 난이도보다 변경 경제성의 문제다. 더 작은 배치와 더 짧은 피드백 주기로 나누면, 충돌 원인을 더 싸게 찾고 되돌릴 수 있다.

**Recommendations**:
1. 모든 하드닝 작업을 `tidy commit`과 `behavior/policy commit` 두 단계로 분리하고, PR 템플릿에 "이 PR의 tidy 범위"를 명시해 변경 경제성을 관리하라.
2. 날짜 경계/정책 위반처럼 반복되는 혼선은 체크 스크립트에 fixture 기반 시나리오 테스트(경계 시각, 타임존, 롤오버)를 추가해 "문서 규칙 -> 실행 규칙" 전환을 완료하라.
